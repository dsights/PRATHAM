1. System Architecture Overview

Financial fraud is rarely an isolated event. It involves networks of accounts. This architecture uses a Graph Database (Neo4j) combined with a GNN (Graph Neural Network) to detect "Money Mules" and "Synthetic Identities" in milliseconds.

Component Breakdown:

  • API Gateway: Validates and sanitizes transaction requests.
  • Transaction Pipeline: Kafka topics for "Raw Transactions," "Enriched Data," and "Fraud Scores."
  • Feature Engine: Flink jobs calculating rolling window stats (e.g., transaction frequency last 10 mins).
  • Graph Store: Neo4j cluster storing entities (Users, Cards, IPs, Devices) and relationships (Transacted_With, Logged_In_From).
  • GNN Inference: Python-based service using PyTorch Geometric for embedding generation.

2. Problem Statement

Traditional fraud detection relies on "if-then" rules which are easily bypassed by sophisticated gangs.

  • Data Silos: Rule engines often look at one transaction at a time, missing the "big picture" of relationship hops.
  • Speed: Detecting a fraud ring after the money has left the bank is useless; detection must happen "in-flight."
  • Evolving Tactics: Fraudsters change patterns weekly, making static rules obsolete.

3. Engineering Approach to Solve the Problem

  • Graph-Native Features: Instead of just "Transaction Amount," we extract "Degree Centrality" (how many accounts is this IP connected to?) and "PageRank" of the account.
  • In-Memory Graph Processing: Use a read-replica of the graph in memory for sub-10ms traversal during the transaction flow.
  • Asynchronous Enrichment: Transactions are authorized based on high-level risk, while a secondary "Deep Audit" happens in parallel to freeze funds if high-risk graph patterns are found.

4. The AI Advantage Used

  • GNN (Graph Neural Networks): These models learn "neighborhood" embeddings. A user might look normal, but if they are 2 hops away from a known fraudster, the GNN increases the risk score based on the network topology.
  • Self-Supervised Learning: Since labels (confirmed fraud) are rare, we use contrastive learning to group "unusual" behaviors together for human review.
